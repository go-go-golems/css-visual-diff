You do not need prior knowledge of css-visual-diff, the Pyxis project, or visual regression testing. This guide explains everything from scratch.

## 1. What Problem Are We Solving?

When a developer changes CSS or component markup in a web application, they need to know whether the visual output still matches the intended design. This is called **visual regression testing**. The computer takes screenshots of two versions of the same page (a "prototype" reference and a "React implementation") and measures the pixel differences.

But measuring differences is only half the job. The other half is **human judgment**: a person needs to look at the screenshots, decide if each difference is acceptable, write notes about what to fix or accept, and pass those notes to someone who will change the code.

Today, css-visual-diff produces a folder of PNG screenshots, JSON metadata files, and a summary. A Python script converts these into a single static HTML page with basic image grids and textareas. That page works, but it has critical limitations:

- **Notes disappear on reload.** The textareas have no persistence. If you close the tab, your review is gone.
- **No interactive comparison modes.** You see three static images (prototype, React, diff) in a grid. You cannot overlay them, slide between them, or zoom into differences.
- **No annotation system.** You can type notes in a textarea, but you cannot pin a comment to a specific spot on the screenshot.
- **No status tracking.** The tool classifies each section as "accepted", "review", "tune-required", or "major-mismatch" based on pixel percentages, but there is no way for the human reviewer to override that with their own verdict ("I accept this", "this needs work", "won't fix").
- **Not embedded in the tool.** The review page is generated by a separate Python script. It is not part of the css-visual-diff binary.

We are building a **React single-page application (SPA)** that solves all of these problems. It will be compiled to static files, embedded into the Go binary, and served via a `css-visual-diff serve` command.

## 2. What Is css-visual-diff?

`css-visual-diff` is a Go command-line tool that compares rendered HTML/CSS across browser targets. It lives in the repository at:

```
/home/manuel/code/wesen/corporate-headquarters/css-visual-diff
```

The tool has several subcommands:

| Command | What it does |
|---|---|
| `run` | Run capture and analysis modes from a YAML config file |
| `inspect` | Inspect one side of a config (take screenshots, extract CSS) |
| `compare` | Compare a single element between two URLs (screenshots + CSS + pixel diff) |
| `llm-review` | Run compare and ask an LLM for a multimodal review |
| `chromedp-probe` | Run a minimal browser probe against a URL |
| `verbs` | Run user-defined JavaScript workflow verbs from a repository |

The tool uses **chromedp** (a Go library that drives Chrome/Chromium via the Chrome DevTools Protocol) to navigate pages, wait for rendering, take screenshots, and extract computed CSS properties.

### Key concepts

- **Prototype**: The static reference HTML page (the "ground truth" design). Usually served from a local static server.
- **React implementation**: The actual React-rendered page, typically running in Storybook.
- **Visual spec**: A YAML file that declares which pages to compare, which selectors to crop, viewport sizes, and threshold policies.
- **Run**: One execution of css-visual-diff against a spec, producing PNGs and JSON.
- **Section**: One cropped comparison within a page. A page may have multiple sections (e.g., "content", "header", "mailing-list").
- **Classification**: A label computed from the pixel-change percentage: `accepted`, `review`, `tune-required`, or `major-mismatch`.

### The YAML config format

A typical visual spec looks like this (simplified):

```yaml
name: public-pages
targets:
  original:
    base: http://localhost:7070
  react:
    base: http://localhost:6007

pages:
  about:
    original: /standalone/public/about.html
    react: /iframe.html?id=public-site-pages-about--desktop&viewMode=story
    sections:
      content:
        originalSelector: "[data-page='about']"
        reactSelector: "[data-page='about']"

policy:
  bands:
    - name: accepted
      maxChangedPercent: 0.5
    - name: review
      maxChangedPercent: 10
    - name: tune-required
      maxChangedPercent: 30
    - name: major-mismatch
      maxChangedPercent: 100
```

The `policy.bands` define the classification thresholds. If a comparison shows 7% changed pixels, it falls into the `review` band (between 0.5% and 10%).

## 3. The Current Workflow: How Reviews Work Today

Here is the step-by-step process that a human reviewer currently follows:

```text
┌──────────────────────────────┐
│ 1. Run css-visual-diff        │
│    compare spec → outDir      │
│    produces PNGs + JSONs      │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 2. Run Python generator       │
│    reads summary JSON          │
│    copies artifacts            │
│    writes index.html           │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 3. Serve with Python HTTP     │
│    python3 -m http.server      │
│    on port 8097                │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 4. Human reviews cards         │
│    looks at images             │
│    types notes (lost on reload)│
│    copies YAML to clipboard    │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 5. Paste notes into ticket     │
│    or PR or chat               │
└──────────────────────────────┘
```

### Problems with this workflow

1. **Two separate tools.** The reviewer runs a Go CLI to capture, then a Python script to generate the page. The new system should be one tool.

2. **Notes are ephemeral.** The static HTML uses plain `<textarea>` elements with no storage. Refreshing the page loses everything. The new system uses `localStorage` to persist notes and status decisions.

3. **No interactive image comparison.** The current page shows three images in a grid. You cannot overlay them, slide between them, or zoom in. The mock JSX (`diff-review.jsx`) demonstrates four view modes that the new app will implement.

4. **No annotation pins.** The current page has no way to mark a specific location on a screenshot and attach a comment. The mock shows a pin-drop system with comment types (issue, note, question, praise).

5. **No status tracking.** The tool computes classification from pixels, but the human cannot record their own decision (accept, needs-work, fixed, won't-fix) separately from the computed classification.

6. **Not portable.** The static HTML references files in `/tmp` with absolute paths. The new app should work with relative paths so the bundle is self-contained.



## 4. The New Thing We Are Building

We are building a **React single-page application** called the **css-visual-diff Review Site**. It replaces the static HTML generator with a proper web app that provides:

1. **Interactive image comparison** with four view modes:
   - **Side-by-side**: prototype and React images next to each other.
   - **Overlay**: stack both images with adjustable opacity; hold spacebar to flash.
   - **Slider**: drag a handle left/right to reveal prototype or React via CSS `clip-path`.
   - **Diff only**: show only the pixels that changed (from `diff_only.png`).

2. **Pin-drop annotations** where the reviewer clicks on any image to drop a numbered comment pin. Each pin has a type (issue, note, question, praise) and a text field.

3. **Review status tracking** where each section gets a human verdict: `unreviewed`, `accepted`, `needs-work`, `fixed`, or `wont-fix`. This is stored in the browser's `localStorage` keyed by `(page, section, run-id)`.

4. **CSS diff sidebar** that shows the computed style differences between prototype and React, extracted from `compare.json`.

5. **Markdown + YAML export** that formats the review notes, pin comments, and computed metadata into a clipboard-ready block that can be pasted into an issue, PR comment, or LLM prompt.

6. **Embedded serving** where the built SPA is embedded into the Go binary via `go:embed` and served by a new `css-visual-diff serve` subcommand.

### Technology choices

| Layer | Technology | Why |
|---|---|---|
| UI framework | React 18+ | The mock JSX is already React. Team is familiar. |
| State management | Redux Toolkit (RTK) | Structured state for review data, filters, view modes, comments, and local storage sync. |
| Build tool | Vite | Fast builds, good for embedding into Go binaries. |
| Styling | Tailwind CSS | The mock already uses Tailwind utility classes. |
| Icons | Lucide React | The mock already imports from lucide-react. |
| Language | TypeScript | Type safety for the data model (compare.json, summary.json). |
| Persistence | Browser localStorage | No backend needed for v1. Keyed by run-id + page + section. |
| Server | Go binary with `go:embed` | Single binary distribution. No Node.js runtime needed at serve time. |

### How the pieces connect

1. The developer runs `css-visual-diff run` or `css-visual-diff compare` to produce a run directory full of artifacts.

2. The developer runs `css-visual-diff serve --data-dir /tmp/run-dir --port 8097`.

3. The Go binary starts an HTTP server on port 8097 that:
   - Serves the embedded React SPA files at `/` (index.html, JS bundles, CSS).
   - Serves artifact files (PNGs, JSON) at `/artifacts/...` by reading from `--data-dir`.
   - Serves the summary JSON at `/api/manifest` by reading `summary.json` from `--data-dir` (or the suite summary JSON specified via `--summary`).

4. The React SPA loads in the browser. On startup it fetches `/api/manifest` to get the list of review cards (pages, sections, classifications, artifact paths).

5. For each card the user expands or views, the SPA fetches the individual `compare.json` file lazily from `/artifacts/page/section/compare.json`.

6. The user reviews cards, writes notes, drops pins, sets statuses. All feedback is persisted to `localStorage`.

7. The user clicks "Send to LLM" to export a markdown+yaml block to the clipboard.

### 6.2 The summary JSON

The summary JSON is the **entry point** for the review site. It is produced by running `css-visual-diff verbs ... --summary --output json` and contains one row per page/section combination.

**File location**: specified by the user (e.g., `/tmp/pyxis-public-pages-final-sweep.json`).

**Shape**: It is either a list with one element containing `rows`, or a plain object with `rows`:

```json
[

    "classificationCounts": {
      "review": 7,
      "tune-required": 6
    },
    "pageCount": 5,
    "maxChangedPercent": 11.605,
    "policy": {
      "ok": false,
      "worstClassification": "tune-required"
    },
    "rows": [
      {
        "page": "shows",
        "section": "content",
        "classification": "tune-required",
        "changedPercent": 11.605,
        "changedPixels": 197517,
        "totalPixels": 1702000,
        "diffOnlyPath": "/tmp/.../shows/artifacts/content/diff_only.png",
        "diffComparisonPath": "/tmp/.../shows/artifacts/content/diff_comparison.png",
        "leftRegionPath": "/tmp/.../shows/artifacts/content/left_region.png",
        "rightRegionPath": "/tmp/.../shows/artifacts/content/right_region.png",
        "artifactJson": "/tmp/.../shows/artifacts/content/compare.json",
```

**Key fields in each row**:

| Field | Type | Meaning |
|---|---|---|
| `page` | string | Page name (e.g., "shows", "about", "book") |
| `section` | string | Section within the page (e.g., "content", "header", "page") |
| `classification` | string | Computed label: "accepted", "review", "tune-required", "major-mismatch" |
| `changedPercent` | number | Percentage of pixels that differ |
| `changedPixels` | number | Raw count of changed pixels |
| `diffOnlyPath` | string | Absolute path to `diff_only.png` |
| `leftRegionPath` | string | Absolute path to prototype screenshot |
| `rightRegionPath` | string | Absolute path to React screenshot |
| `artifactJson` | string | Absolute path to `compare.json` |
| `leftSelector` | string | CSS selector used for prototype cropping |
| `rightSelector` | string | CSS selector used for React cropping |
| `styleDiffs` | array | List of changed CSS properties with left/right values |
| `attributeDiffs` | array | List of changed HTML attributes |
| `bounds` | object | Bounding box comparison (left vs right, delta) |

### 6.4 The PNG artifacts

Each section produces up to four PNG files:

| File | What it shows |
|---|---|
| `left_region.png` | The cropped screenshot of the prototype element |
| `right_region.png` | The cropped screenshot of the React element |
| `diff_only.png` | Only the pixels that differ, highlighted in red/magenta |
| `diff_comparison.png` | A side-by-side triptych (left, diff, right) |

These images have **normalized dimensions** — both prototype and React screenshots are resized to the same dimensions before comparison. The `pixel.normalizedWidth` and `pixel.normalizedHeight` fields record these dimensions.

## 10. State Management with Redux Toolkit (RTK)

Redux Toolkit (RTK) is the standard way to manage state in modern React apps. It provides a clean API for defining state slices, reducers, and async actions.

### Why RTK instead of plain React state?

The review site has several pieces of state that need to be shared across components:

- The **loaded cards** (from the manifest API) are needed by CardList, FilterToolbar, and Header.
- The **active card** selection affects the Sidebar, Header, and ImageCanvas.
- The **comment pins** are edited in the Sidebar but rendered on the Canvas.
- The **review status and notes** must be persisted to localStorage whenever they change.

Using RTK keeps all of this in one place with clear update logic.

### 10.1 Store configuration

```typescript

  reducer: {
    cards: cardsReducer,
    view: viewReducer,
```

### 10.2 Cards slice

The cards slice holds the loaded summary data and filter state.

```typescript
// store/slices/cardsSlice.ts
import { createSlice, createAsyncThunk, PayloadAction } from '@reduxjs/toolkit';
import { SummaryRow, SuiteSummary, SummaryPayload } from '../../types/summary';

interface CardsState {
  rows: SummaryRow[];
  loading: boolean;
  error: string | null;
  filter: {
    classification: string | null;   // null = show all
    status: string | null;           // null = show all
    search: string;                  // text search on page/section
  };
  selectedCardIndex: number;
}

const initialState: CardsState = {
  rows: [],
  loading: false,
  error: null,
  filter: { classification: null, status: null, search: '' },
  selectedCardIndex: 0,
};

// Async thunk to fetch /api/manifest
export const fetchManifest = createAsyncThunk(
  'cards/fetchManifest',
  async () => {
    const response = await fetch('/api/manifest');
    if (!response.ok) throw new Error(`Failed to load manifest: ${response.status}`);
    const payload: SummaryPayload = await response.json();
    // Normalize: accept both [SuiteSummary] and SuiteSummary
    const summary: SuiteSummary = Array.isArray(payload) ? payload[0] : payload;
    return summary.rows;
  }
);

const cardsSlice = createSlice({
  name: 'cards',
  initialState,
  reducers: {
    setClassificationFilter(state, action: PayloadAction<string | null>) {
      state.filter.classification = action.payload;
    },
    setStatusFilter(state, action: PayloadAction<string | null>) {
      state.filter.status = action.payload;
    },
    setSearch(state, action: PayloadAction<string>) {
      state.filter.search = action.payload;
    },
    selectCard(state, action: PayloadAction<number>) {
      state.selectedCardIndex = action.payload;
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchManifest.pending, (state) => { state.loading = true; })
      .addCase(fetchManifest.fulfilled, (state, action) => {
        state.loading = false;
        state.rows = action.payload;
      })
      .addCase(fetchManifest.rejected, (state, action) => {
        state.loading = false;
        state.error = action.error.message || 'Failed to load';
      });
  },
});
```

### 10.6 localStorage sync middleware

```typescript
// store/middleware/localStorageSync.ts
import { Middleware } from '@reduxjs/toolkit';

// This middleware intercepts every action that modifies the review slice
// and writes the updated state to localStorage.

export const localStorageSync: Middleware = (store) => (next) => (action) => {
  const result = next(action);

  // After any review-related action, persist to localStorage
  if (action.type.startsWith('review/')) {
    const state = store.getState();
    const reviewState: RunReviewState = {
      runId: state.review.runId,
      cards: state.review.cards,
      updatedAt: new Date().toISOString(),
    };
    const key = `cssvd-review-${state.review.runId}`;
    localStorage.setItem(key, JSON.stringify(reviewState));
  }

  return result;
};
```

- Toggle "difference blend" mode.

```text
│                        [diff blend]     │
│                        hold ␣ to flash  │

│                                         │
│  Layer stack:                           │
│  ┌─────────────────┐                    │
│  │ prototype (base) │                   │
│  │  ┌─────────────┐ │                   │
│  │  │ react (top)  │ │  ← opacity = 50% │
│  │  └─────────────┘ │                   │
│  └─────────────────┘                    │
└─────────────────────────────────────────┘
```

Pseudocode:
```typescript
function ViewModeOverlay({ leftUrl, rightUrl }: Props) {
  const [opacity, setOpacity] = useState(50);
  const [blend, setBlend] = useState<'normal' | 'difference'>('normal');
  const [swap, setSwap] = useState(false);  // spacebar flash

  useEffect(() => {
    // Spacebar listener for flash
    const down = (e) => { if (e.code === 'Space') { e.preventDefault(); setSwap(true); } };
    const up = (e) => { if (e.code === 'Space') setSwap(false); };
    window.addEventListener('keydown', down);
    window.addEventListener('keyup', up);
    return () => { window.removeEventListener('keydown', down); window.removeEventListener('keyup', up); };
  }, []);

  const effectiveOpacity = swap ? (opacity > 50 ? 0 : 100) : opacity;

  return (
    <div className="relative">
      <img src={leftUrl} className="block" />
      <img
        src={rightUrl}
        className="absolute inset-0"
        style={{
          opacity: effectiveOpacity / 100,
          mixBlendMode: blend === 'difference' ? 'difference' : 'normal',
        }}
      />
    </div>
  );
}
```

### 12.3 Slider mode

A vertical divider that the user drags left/right. The prototype is revealed on the left of the divider, the React on the right, using CSS `clip-path`.

```text
┌────────────────────────────────────────┐
│                      │                 │
│   prototype          │ react           │
│   (left of handle)   │ (right of handle)│
│                      │                 │
│                      ║ ← draggable     │
│                      ║   handle        │
│                      │                 │
└────────────────────────────────────────┘
```

Pseudocode:
```typescript
function ViewModeSlider({ leftUrl, rightUrl }: Props) {
  const [position, setPosition] = useState(50);  // 0 = all A, 100 = all B
  const containerRef = useRef<HTMLDivElement>(null);

  // Mouse drag handler (simplified)
  const handleMouseDown = () => {
    const handleMouseMove = (e: MouseEvent) => {
      const rect = containerRef.current!.getBoundingClientRect();
      const pos = ((e.clientX - rect.left) / rect.width) * 100;
      setPosition(Math.max(0, Math.min(100, pos)));
    };
    const handleMouseUp = () => {
      window.removeEventListener('mousemove', handleMouseMove);
      window.removeEventListener('mouseup', handleMouseUp);
    };
    window.addEventListener('mousemove', handleMouseMove);
    window.addEventListener('mouseup', handleMouseUp);
  };

  return (
    <div ref={containerRef} className="relative">
      {/* Prototype: clipped to left portion */}
      <img src={leftUrl} style={{ clipPath: `inset(0 ${100 - position}% 0 0)` }} />

      {/* React: clipped to right portion */}
      <img
        src={rightUrl}
        className="absolute inset-0"
        style={{ clipPath: `inset(0 0 0 ${position}%)` }}
      />

      {/* Draggable handle */}
      <div
        className="absolute top-0 bottom-0"
        style={{ left: `${position}%` }}
        onMouseDown={handleMouseDown}
      >
        <div className="w-px h-full bg-black/40" />
        <div className="absolute top-1/2 -translate-y-1/2 -translate-x-1/2
                        w-7 h-7 rounded-full bg-black text-white
                        flex items-center justify-center cursor-ew-resize">
          ⋮⋮
        </div>
      </div>

}
```

## 13. Comment Pins: Click-to-Annotate

Comment pins let the reviewer click on any spot in the image area and drop a numbered marker. Each pin records:
- Its position as percentage coordinates (x% and y% of the image width/height).
- Which "side" it belongs to: `left` (prototype), `right` (react), or `merged` (overlay/slider/diff views).
- A type: `issue` (red), `note` (amber), `question` (teal), or `praise` (plum).

```typescript
function ImageCanvas({ children, commentMode, onAddComment, pins, ... }) {
  const containerRef = useRef<HTMLDivElement>(null);

  const handleClick = (e: React.MouseEvent) => {
    if (!commentMode) return;
    const rect = containerRef.current!.getBoundingClientRect();
    const x = ((e.clientX - rect.left) / rect.width) * 100;
    const y = ((e.clientY - rect.top) / rect.height) * 100;
    onAddComment({ x, y, side: currentSide });
  };

  return (
    <div
      ref={containerRef}
      onClick={handleClick}
      className={commentMode ? 'cursor-crosshair' : ''}
    >
      {children}
      {pins.map((pin, i) => (
        <CommentPin key={pin.id} comment={pin} index={i} ... />
      ))}
    </div>
  );
}
```

## 15. Review Status and Local Storage

### 15.1 Review status values

The human reviewer assigns a status to each card. This is **separate from** the computed classification:

| Status | Meaning | When to use |
|---|---|---|
| `unreviewed` | No human has looked at this yet | Default when the page loads |
| `accepted` | The reviewer says this is fine | Even if classification is "tune-required" |
| `needs-work` | The reviewer wants a developer to fix something | When the diff shows real problems |
| `fixed` | A developer has made changes and they look correct | After implementing feedback |
| `wont-fix` | Acknowledged difference that should remain as-is | Intentional design divergence |

### 15.2 The important distinction

A card can be classified as `tune-required` (11% pixel difference) but accepted by the human reviewer. This happened in the real Pyxis project with the Shows page. The tool said "tune-required" but Manuel said "this is fine, move on." The review site must record both values independently.

### 15.3 How localStorage persistence works

When the page loads:

```typescript

  const key = `cssvd-review-${runId}`;
  const saved = localStorage.getItem(key);
  if (saved) {
    const parsed: RunReviewState = JSON.parse(saved);

```

When any review action fires (setStatus, setNote, addPin, etc.):

```typescript

```

When the user opens the page again:

```text
1. React app loads
2. fetchManifest() loads /api/manifest → gets rows
3. For each row, check if localStorage has a saved review
4. If yes, restore status, note, and comments
5. Reviewer picks up where they left off
```

// utils/export.ts
function buildExportMarkdown(
  row: SummaryRow,
  compareData: CompareData | null,
  review: CardReview,
): string {
  const parts: string[] = [];

  // Header
  parts.push(`# Visual review — \`${row.page} / ${row.section}\``);
  parts.push('');
  parts.push(`**Classification:** ${row.classification} · ${row.changedPercent.toFixed(2)}% changed`);
  parts.push(`**Status:** ${review.status}`);
  if (compareData) {
    parts.push(`**Bounds delta:** Δheight ${compareData.bounds.delta.height}px`);
  }
  parts.push('');

  // General observation
  if (review.note) {
    parts.push('## General observation');
    parts.push('');
    parts.push(review.note);
    parts.push('');
  }

  // Pin-drop comments
  if (review.comments.length > 0) {
    parts.push('## Pin-drop comments');
    parts.push('');
    review.comments.forEach((c, i) => {
      const sideLabel = c.side === 'left' ? 'prototype' : c.side === 'right' ? 'react' : 'merged';
      parts.push(`${i + 1}. **[${c.type}]** _${sideLabel} @ ${c.x.toFixed(0)}%, ${c.y.toFixed(0)}%_ — ${c.text}`);
    });
    parts.push('');
  }

  // Style diffs
  if (row.styleDiffs && row.styleDiffs.length > 0) {
    parts.push('## Computed style diffs');
    parts.push('');
    row.styleDiffs.forEach((s) => {
      parts.push(`- \`${s.property}\`: \`${s.left}\` → \`${s.right}\``);
    });
    parts.push('');
  }

  // YAML block
  parts.push('---');
  parts.push('');
  parts.push('```yaml');
  parts.push(yamlStringify({
    page: row.page,
    section: row.section,
    classification: row.classification,
    changedPercent: row.changedPercent,
    bounds: compareData?.bounds ? {
      delta: compareData.bounds.delta,
      left: compareData.bounds.left,
      right: compareData.bounds.right,
    } : undefined,
    pixel: compareData?.pixel ? {
      changedPixels: compareData.pixel.changedPixels,
      totalPixels: compareData.pixel.totalPixels,
      threshold: compareData.pixel.threshold,
    } : undefined,
    review: {
      status: review.status,
      generalNote: review.note,
      comments: review.comments.map((c, i) => ({
        id: i + 1,
        type: c.type,
        side: c.side,
        position: { x: c.x, y: c.y },
        text: c.text,
      })),
    },
  }));
  parts.push('```');

  return parts.join('\n');
}

## 17. The Go Integration: `serve` Command

The `serve` command is a new subcommand added to the `css-visual-diff` binary. It starts an HTTP server that:
1. Serves the embedded React SPA (built with Vite, embedded with `go:embed`).
2. Serves artifact files from a user-specified directory.
3. Provides a `/api/manifest` endpoint that returns the summary JSON.

### 17.1 Command interface

```bash
css-visual-diff serve --data-dir /tmp/pyxis-public-pages-final-sweep --port 8097

# Or with an explicit summary file:
css-visual-diff serve \
  --data-dir /tmp/pyxis-public-pages-final-sweep \
  --summary /tmp/pyxis-public-pages-final-sweep.json \
  --port 8097
```

### 17.2 Flags

| Flag | Type | Default | Description |
|---|---|---|---|
| `--data-dir` | string | required | Path to the css-visual-diff run output directory |
| `--summary` | string | "" | Explicit path to summary JSON (if not in data-dir) |
| `--port` | int | 8097 | HTTP server port |
| `--host` | string | "127.0.0.1" | Bind address |
| `--open` | bool | false | Open browser automatically |

### 17.3 HTTP routing

```text
GET /                         → serve embedded index.html
GET /assets/*                 → serve embedded JS/CSS bundles
GET /artifacts/*              → serve files from --data-dir
GET /api/manifest             → return summary JSON content
GET /api/compare?page=X&section=Y → return specific compare.json
```

### 17.4 Implementation pseudocode (Go)

```go
// internal/cssvisualdiff/serve/serve.go
package serve

import (
    "embed"
    "encoding/json"
    "io/fs"
    "net/http"
    "os"
    "path/filepath"
)

//go:embed all:spa
var spaFS embed.FS

type Server struct {
    dataDir    string
    summary    string
    host       string
    port       int
}

func NewServer(opts Options) *Server { ... }

func (s *Server) ListenAndServe() error {
    mux := http.NewServeMux()

    // 1. API endpoints
    mux.HandleFunc("/api/manifest", s.handleManifest)
    mux.HandleFunc("/api/compare", s.handleCompare)

    // 2. Artifact files from data-dir
    mux.Handle("/artifacts/", http.StripPrefix("/artifacts/",
        http.FileServer(http.Dir(s.dataDir))))

    // 3. SPA fallback (embedded)
    spaSub, _ := fs.Sub(spaFS, "spa")
    fileServer := http.FileServer(http.FS(spaSub))
    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        // Try embedded file first
        if _, err := spaSub.Open(r.URL.Path); err == nil {
            fileServer.ServeHTTP(w, r)
            return
        }
        // Fallback to index.html for SPA routing
        r.URL.Path = "/"
        fileServer.ServeHTTP(w, r)
    })

    addr := fmt.Sprintf("%s:%d", s.host, s.port)
    log.Printf("Serving review site at http://%s", addr)
    return http.ListenAndServe(addr, mux)
}

func (s *Server) handleManifest(w http.ResponseWriter, r *http.Request) {
    summaryPath := s.summary
    if summaryPath == "" {
        // Try to find summary.json in data-dir
        summaryPath = filepath.Join(s.dataDir, "summary.json")
    }
    data, err := os.ReadFile(summaryPath)
    if err != nil {
        http.Error(w, "summary not found", http.StatusNotFound)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    w.Write(data)
}

func (s *Server) handleCompare(w http.ResponseWriter, r *http.Request) {
    page := r.URL.Query().Get("page")
    section := r.URL.Query().Get("section")
    path := filepath.Join(s.dataDir, page, "artifacts", section, "compare.json")
    data, err := os.ReadFile(path)
    if err != nil {
        http.Error(w, "compare.json not found", http.StatusNotFound)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    w.Write(data)
}
```

### 17.5 Path rewriting

The Go server needs to rewrite the absolute paths in the summary JSON into relative `/artifacts/` URLs. The summary rows contain paths like `/tmp/pyxis-.../shows/artifacts/content/diff_only.png`. The server strips the data-dir prefix and serves them under `/artifacts/`.

Alternatively, the React app can do this path rewriting client-side:

```typescript
// utils/paths.ts
function toArtifactUrl(absolutePath: string, dataDirPrefix?: string): string {
  // Strategy: extract page/section from path pattern
  // /tmp/.../shows/artifacts/content/diff_only.png → /artifacts/shows/content/diff_only.png
  const match = absolutePath.match(/\/([^/]+)\/artifacts\/([^/]+)\/([^/]+)$/);
  if (match) {
    return `/artifacts/${match[1]}/${match[2]}/${match[3]}`;
  }
  return absolutePath;
}
```

```typescript
const selectFilteredCards = (state: RootState) => {
  const { rows, filter } = state.cards;
  return rows.filter(row => {
    if (filter.classification && row.classification !== filter.classification) return false;
    if (filter.search && !`${row.page}/${row.section}`.includes(filter.search)) return false;
    // Status filter needs review state
    if (filter.status) {

  });
```
